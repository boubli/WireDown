# WireDown Architectural Documentation

## 1. High-Level Architecture & Concept

### The Decoupling Philosophy
Historically, network security sensors built for micro-controllers (like the ESP32) tightly coupled their business logic with hardware-specific APIs. WireDown breaks this paradigm by implementing a **Hardware Abstraction Layer (HAL)**. 

By completely decoupling the lowest layers of the OSI model (Layer 2 frame capture and injection) from the security analytics, the "Core Engine" becomes a pure state machine. It analyzes byte arrays and returns JSON alerts. This "Write once, run anywhere" philosophy ensures that complex detection logic (like KRACK or ARP spoofing) is maintained in exactly one place, yet can be compiled natively for entirely different architectures.

### The Three Deployment Environments
1. **ESP32 (Physical Edge):** Compiles via the Arduino/PlatformIO toolchain. Acts as a stealthy, physical rogue AP detector or physical honeypot on a localized VLAN.
2. **LXC (Virtualized Edge - Proxmox):** Compiles as a native Linux binary inside a lightweight Linux Container. Attaches directly to virtual bridges (e.g., `vmbr0`) to monitor intra-VM traffic with near-zero overhead.
3. **Virtual Machine (Standard Linux):** Compiles as a native binary deployed via Cloud-Init on full OS instances (like Ubuntu). Best for isolated, heavily sandboxed environments.

## 2. The Core Engine (`core/`)

The Core Engine is entirely platform-agnostic C++ code. It relies on three abstract interfaces to interact with the outside world:
*   **`INetworkInterface.h`**: OSI Layer 2 abstraction. Defines how to sniff packets (`startPromiscuous`) and how to forge them (`injectFrame`).
*   **`ITransport.h`**: Telemetry abstraction. Defines how to send JSON alerts (`send`) and receive commands.
*   **`ISystem.h`**: OS-level abstraction for clocks (`getMillis`) and memory profiling (`getFreeMemory`).

### Data Flow & Analytics (`Engine.cpp`)
All raw packets from the `INetworkInterface` funnel into a single entrypoint: `Engine::processPacket(const uint8_t* buffer, ...)`. The engine dissects the 802.11 MAC headers and (if applicable) the LLC/SNAP headers.

*   **Deduplication & Device Tracking:** To prevent flooding the backend with thousands of events per second, the engine maintains a ring-buffer (`seenDevices`) of MAC addresses and a `lastSeen` timestamp. Devices are only reported once per 10-second window unless they are actively attacking.
*   **ARP Spoofing Detection:** By filtering for EtherType `0x0806` (ARP), the engine maintains a localized `arpTable` mapping Sender IPs to MAC addresses. If an ARP Reply claims an IP is at a *different* MAC than previously recorded, it immediately triggers an `arp_spoof` alert.
*   **Deauth / Disassoc Floods:** The engine inspects 802.11 Management Frames (Type `0x00`). If Subtypes `0x0C` (Deauth) or `0x0A` (Disassoc) exceed 10 frames within a rolling 1-second window, it triggers a flood alert.
*   **KRACK Attack Detection:** The engine looks for EAPOL frames (EtherType `0x888E`). By parsing the Key Information fields, it identifies "Message 3" of the WPA2 4-way handshake (Install and Ack bits set). If the engine detects that Message 3 is transmitted 3 or more times to the same client within 30 seconds, it indicates a Key Reinstallation Attack (KRACK).

## 3. The Linux Platform Adapter (`platforms/linux/`)

This platform translates the Core Engine's abstract requests into native Linux Kernel system calls.

### `LinuxNetwork.cpp` (AF_PACKET)
To sniff raw frames natively on Linux, the adapter bypasses standard TCP/UDP stacks and uses `AF_PACKET`.
*   **Capture:** It calls `socket(AF_PACKET, SOCK_RAW, htons(ETH_P_ALL))`. This instructs the kernel to pass every single packet on the interface (including MAC headers) directly to user-space. A dedicated POSIX thread (`pthread`) constantly polls `recvfrom()` and passes the byte array to the Core Engine.
*   **Promiscuous Mode:** It uses `setsockopt` with `PACKET_ADD_MEMBERSHIP` and `PACKET_MR_PROMISC` to force the network interface controller (NIC) to capture packets not destined for its own MAC address.
*   **Injection:** To forge responses (like targeted Deauths), it constructs a `sockaddr_ll` struct and pushes raw byte arrays back onto the wire using `sendto()`.

### `LinuxTransport.cpp`
Implements a standard non-blocking POSIX TCP Socket (`<sys/socket.h>`). It wraps the JSON alerts generated by the core engine into simple string payloads and streams them to the centralized backend.

## 4. The ESP32 Platform Adapter (`platforms/esp32/`)

This platform translates the Core Engine's abstract requests into Espressif SDK (ESP-IDF) functions.

### `Esp32Network.cpp`
*   **Capture:** Relies on the `esp_wifi_set_promiscuous_rx_cb()` function. The ESP32 WiFi radio intercepts 802.11 frames and fires the callback, passing a `wifi_promiscuous_pkt_t` struct directly to the Core Engine.
*   **Injection:** Uses `esp_wifi_80211_tx()` to directly transmit raw 802.11 frames (e.g., forging an AP's BSSID to drop an attacker off the network).

### `Esp32Transport.cpp`
Uses the `ArduinoWebsockets` library to establish a persistent, bidirectional WebSocket connection (`ws://`) to the Python backend. This allows real-time streaming of alerts and immediate reception of active response commands (like `/isolate`).

## 5. Deployment & Automation Lifecycle

WireDown utilizes infrastructure-as-code principles to deploy the Linux sensor autonomously in virtualized environments.

### Proxmox LXC Automation (`proxmox-lxc.sh`)
When executed on the Proxmox host, this script:
1.  Downloads a Debian template and creates an unprivileged container.
2.  **Capability Injection:** Appends `lxc.cap.keep: net_raw net_admin` to the container's config file. By default, Linux containers isolate network capabilities to prevent container-to-host breakouts. The `net_raw` capability is strictly required for the container to open the `SOCK_RAW` socket used by `AF_PACKET`. Without it, `LinuxNetwork.cpp` would be denied permission to sniff.
3.  **In-Container Compilation:** Drops into the container via `pct exec` to install compilers (`build-essential`, `cmake`), builds the C++ Core Engine natively, and dynamically writes a `systemd` unit file to `/etc/systemd/system/wiredown.service` so the sensor runs persistently in the background.

### Proxmox VM Automation (`proxmox-vm.sh`)
Virtual machines require a different approach. The script generates an Ubuntu Cloud image and utilizes **Cloud-Init** to automate the setup:
1.  It generates a `.yml` snippet on the Proxmox host containing `packages` (`cmake`, `build-essential`) and `runcmd` directives.
2.  During the Ubuntu VM's very first boot, the Cloud-Init agent reads the snippet, autonomously compiles the C++ codebase, creates the `systemd` service, and initializes the Python backend via Docker Compose—resulting in a zero-touch, fully configured VM.
